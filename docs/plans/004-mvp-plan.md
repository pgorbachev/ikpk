# План MVP: static-first запуск с сохранением SEO

**Дата:** 2026-07-11
**Статус:** Предложен
**Приоритет:** P0
**Основание:** синтез анализа текущего состояния `web/`, SEO-аудита (`ikpk_seo_audit_report.md`), рекомендаций маркетолога и архитектурной документации (`docs/architecture/`); проверен независимым ревью полноты.

**Стратегическое решение:** запускаем **static-first MVP** — статический Astro-билд на российском VPS, без Strapi и без собственного бэкенда. Каналы заявок — через уже используемый Bitrix24 (CRM-формы + чат-виджет). Главный риск проекта — не отсутствие фич, а потеря остатков SEO-трафика при миграции 254 URL (сам аудит рекомендовал не делать rebuild), поэтому SEO-миграционный пакет — обязательная часть Definition of Done, а не опция. Из рекомендаций маркетолога в MVP входят только дешёвые точечные правки с прямым влиянием на конверсию; редизайн главной по homepage-draft — post-MVP.

---

## 1) Текущий статус (честно)

1. **Статический прототип готов на ~85–90%**: все 256 страниц собираются, SEO-обвязка (canonical/OG/JSON-LD/sitemap/robots) и богатая тестовая база (vitest + Playwright, 13 файлов) написаны. Но **продукт до go-live готов на ~50–60%**.
2. **Проект заморожен ~3 месяца** (последний коммит 2026-04-13): статусы WBS устарели, 404.astro и serve.sh не закоммичены, последний прогон тестов — failed, README — дефолтный шаблон Astro.
3. **Конверсионный контур не работает**: подписка с `action="#"`, формы записи на семинар на страницах семинаров нет, поиска (FR-05) нет, чат-виджета (FR-13) нет — заявки собирать нечем.
4. **Деплой фактически отсутствует**: текущий таргет GitHub Pages игнорирует `_redirects`/`_headers` (301-редиректы и кеш-заголовки мертвы), VPS не настроен (HTTPS, Nginx, бэкапы, мониторинг), провайдер не зафиксирован.
5. **CI не защищает main**: гоняются только lint/typecheck/audit; unit-, build- и e2e-тесты, Lighthouse-бюджеты в конвейере не запускаются.
6. **Критическая внешняя зависимость**: 100% контентных изображений хотлинкается со storage.yandexcloud.net старого сайта — отключение бакета обнуляет визуал сайта и делает LCP-гейт недостижимым.
7. **Документация противоречива**: migration-runbook.md и observability.md написаны под Netlify/Vercel и противоречат RF-only решению; число страниц гуляет 253/255/256; CMS не подключена (контент заморожен на снапшоте, расписание из 50 записей протухнет первым).

### Не входит (и почему)

| Что | Почему отложено |

| **Strapi + импорт + webhook (FR-11, Phases 2–3)** | Сайт собирается из JSON; 2–10 публикаций/нед первые недели закрывает интерим-процесс через PR. Первый эпик post-MVP с дедлайном ≤6 недель |

| **Лид-магнит (PDF + Telegram-воронка)** | PDF не существует, воронка не спроектирована — блокер на стороне заказчика |

| **Уникализация ~80% шаблонных title/description, длинные H1** | Контентная работа 3–5 дней, аудит — Medium; фиксируем как известный долг |

| **Техдолг TD-1..TD-3, content collections, FAQPage** | Сам tech-debt.md признаёт: актуально при >200 статей; SEO закрыто SSG |

## 3) План по этапам

**Критический путь: Этап 0 → (1 ∥ 2 ∥ 3 ∥ 4) → 5 → 6.** Чистая работа при одном разработчике: **~15–19 рабочих дней до переключения DNS + 14 календарных дней параллельного периода.** Календарная оценка **не включает ожидание заказчика**: этапы 4–6 жёстко зависят от его входов (цифры, расписание, доступы, UAT-слот), а проект уже стоял 3 месяца по орг-причинам — поэтому все запросы уходят в день 1 с дедлайнами, у каждого есть fallback, просрочка эскалируется немедленно.

### Этап 0 — Реанимация репо и CI-фундамент — **1.5–2 дня** *(критический путь, блокирует всё)*

Задачи:

- Закоммитить 404.astro, решить судьбу serve.sh; удалить React 19 + @astrojs/react (0 островов — проверено).
- CI PR-гейт: vitest (unit + build по dist), Playwright smoke (desktop+mobile chromium); полный compat (7 проектов) и parity-тесты против живого ikpk.su — в nightly, не в PR (не должны валить CI).
- Lighthouse CI workflow: бюджеты из kpi-validation.md (Perf mobile ≥85, LCP ≤2.5s, TBT ≤200ms, CLS ≤0.1, SEO ≥95), медиана 5 прогонов на 4 шаблонах, fail = блок деплоя; axe-core в Playwright (A11y ≥90).
- Branch protection main: PR-only, required checks; trunk-based, крупное — за feature-флагом.
- Разобрать причину failed в test-results/.last-run.json; актуализировать WBS; переписать README; создать docs/backlog-related-domains.md.
- **Единый список запросов заказчику с дедлайнами (день 1!):**
  - провайдер VPS — **критерий: ДЦ в Москве/СПб** (под TTFB-гейт p75 ≤300ms);
  - **доступ к DNS-панели/регистратору ikpk.su** + инвентаризация текущей зоны (A/AAAA/MX/SPF/TXT/DKIM) — без этого невозможны ни TTL 300s, ни переключение, а смена A-записи не должна тронуть почту;
  - цифры trust-полосы (5000+ vs 14000+), номер лицензии, актуальное расписание;
  - текст/подтверждение политики обработки ПДн (152-ФЗ);
  - доступы к Метрике/Вебмастеру/GSC, слот UAT.

Критерий готовности: PR в main невозможен без зелёных lint/typecheck/unit/build/smoke/LHCI/axe; React отсутствует в зависимостях; сборка 256 страниц проходит; список запросов отправлен заказчику.

### Этап 1 — Инфраструктура: VPS, staging, редиректы как код — **3 дня** *(∥ этапам 2–4)*

Задачи:

- Зафиксировать провайдера (решение владельца, с учётом критерия локации ДЦ); `infra/` в репо: Nginx-конфиги, идемпотентный bootstrap-vps.sh, deploy-web.sh (releases + symlink current, rollback одной командой).
- Nginx: **gzip + brotli обязательны** (явное требование конфига, проверяется curl'ом в критерии готовности).
- Генератор Nginx-map 301 из url_map.csv (скрипт в CI, конфиг коммитится); http→https и www→non-www **в один хоп** (не воспроизводить 2 хопа старого сайта); единая политика trailing slash.
- HTTPS (Certbot + автопродление) для ikpk.su и staging.ikpk.su; security headers (X-Content-Type-Options, X-Frame-Options, Referrer-Policy, Permissions-Policy); Cache-Control: статика immutable, HTML — короткий public max-age (не no-store).
- **CSP — сначала в Content-Security-Policy-Report-Only** и обкатка против всех третьих сторон (Метрика, Mail.ru top, Bitrix24 чат и CRM-формы, YouTube-фасад, Яндекс.Карты); enforce — только на этапе 5 после чистого прогона.
- GitHub Actions: push в main → авто-деплой staging; prod — workflow_dispatch (ручной триггер).
- UptimeRobot (1-мин чек), ежедневный бэкап /var/www + конфигов.

Критерий готовности: staging по HTTPS; validate-urls.ts против staging = 100% из 1022 URL (200/301 в 1 хоп); securityheaders.com — A; gzip/brotli подтверждены curl; повторный bootstrap не ломает сервер; rollback отрепетирован.

### Этап 2 — Миграция изображений — **3–4 дня** *(∥, старт в день 1 — риск-приоритет)*

Картинки живут в двух местах, подходы разные:

- **Легаси-HTML (основной объём, сотни изображений внутри body-HTML из discovery/entities):** `astro:assets`/`<Image>` к HTML-строкам неприменим. Скрипт собирает все URL storage.yandexcloud.net из entities, скачивает ассеты в репо/хранилище; **новый шаг в html-cleaner** рерайтит URL на локальные пути; width/height проставляются при рерайте (по метаданным скачанных файлов) — против CLS; отдача через Nginx с immutable-кешем.
- **Картинки из полей данных** (превью карточек, hero, преподаватели): переключить на astro:assets/`<Image>` (AVIF/WebP, srcset).
- Grep-тест в vitest.build: **0 хотлинков на чужой бакет в dist/** — навсегда в CI.
- Полноценная OG-image 1200×630 вместо /favicon.svg + og:locale/type/site_name.

Критерий готовности: тест «0 внешних image-запросов» зелёный в CI; выборочная проверка 10 контентных страниц (картинки на месте, без CLS); OG валиден в дебаггерах.

### Этап 3 — SEO-пакет: закрыть разрывы аудита — **2 дня** *(∥)*

Задачи:

- Sitemap: все /statyi/* (у старого их не было — не унаследовать), lastmod везде включая корневые; robots.txt: Sitemap + Clean-Param (utm/yclid/gclid); осознанное решение по Disallow: /*.pdf.
- Сироты (аудит #1): внутренние ссылки на все 27 преподавателей (со страниц курсов) и 6 плейлистов; build-тест обхода dist/ по внутренним ссылкам — 0 сирот.
- **Фикс «Links are not crawlable»** (High в Lighthouse SEO): все навигационные элементы — настоящие `<a href>`.
- **BreadcrumbList и на страницах глубины 1** (сейчас только 2+); **удалить избыточные ItemList-блоки навигации** (3–4 на страницу — шум для парсеров).
- Внешние ссылки per domain_strategy.md: «Фото» → rel="nofollow noopener"; grep-тест на 0 ссылок staging.ikpk.su/medshop в dist/.
- Валидация JSON-LD 5 типов скриптом в CI + иерархия H1→H2→H3 на шаблонах статей/расписания/курса; фиксы typo «кинезиолога» и дубля title вебинаров; 404 < 20KB.

Критерий готовности: все проверки в CI зелёные; Rich Results Test чист на 4 шаблонах (добавлен в pre-launch чек-лист kpi-validation.md).

### Этап 4 — Конверсионный минимум — **3.5–4 дня** *(∥, зависит от этапа 0)*

Задачи:

- Pagefind (FR-05): индексация на build, поиск из Header.astro (toggle уже есть), ленивая загрузка — 1 день.
- Bitrix24 Open Lines (FR-13): loader через requestIdleCallback по паттерну Метрики; добавить задачу в WBS — 0.5 дня.
- FR-06: CTA «Записаться» → **embed CRM-формы Bitrix24 на самом сайте** (не редирект на b24-…bitrix24site.ru: уход с домена рвёт цели Метрики и UTM-атрибуцию) на каждой активной записи расписания и странице семинара; UTM-разметка — 1 день.
- FR-07: NewsletterSignup → Bitrix24-форма с чекбоксом согласия ПДн; fallback — скрыть блок (дефолт при отсутствии согласования) — 0.5 дня.
- **Страница политики обработки ПДн (152-ФЗ)**: проверить, есть ли среди 9 статических страниц; если нет — опубликовать (текст согласует заказчик, есть типовые шаблоны); ссылки из всех форм и футера, упоминание Метрики — 0.5 дня.
- **Лёгкие эмбеды**: YouTube-плейлисты через фасад «превью + клик → iframe» (FR-04; заодно снимает вопрос замедления YouTube — плюс задача post-MVP на RUTUBE/VK-зеркала per PRD); Яндекс.Карты на /kontakty — лениво по scroll/click (FR-08) — 0.5 дня.
- Quick-wins главной: офферный H1, кнопка «Записаться» в Header (сквозная), trust-полоса с подтверждёнными цифрами, блок «Ближайшие семинары» из data.ts (сортировка по дате) — 1 день. *(Опора на структуру homepage-draft, но только блоки, не требующие контента заказчика.)*
- Цели Метрики: клик CTA шапки/hero, «Оставить заявку» (submit embed-формы), submit подписки — задокументировать в observability.md.

Критерий готовности: тестовая заявка доходит до Bitrix24 **и фиксируется целью Метрики**; поиск находит семинар с опечаткой 1–2 символа; на сайте 0 форм с action="#"; политика ПДн опубликована и слинкована; LHCI-бюджеты не просели после виджетов и Pagefind; e2e «CTA шапки → расписание → форма заявки» зелёный.

### Этап 5 — Качественные гейты, контент-freeze, runbook — **2.5–3 дня** *(критический путь, после 1–4)*

Задачи:

- Полный прогон гейтов на staging (реальный Nginx/gzip/brotli): LHCI медиана 5 прогонов × 4 шаблона, axe, compat 7 проектов; **буфер на фикс найденного — основная часть оценки**.
- **CSP: разбор report-only отчётов → перевод в enforce**; все виджеты (Метрика, Bitrix24 чат/формы, YouTube-фасад, Карты) работают под enforce.
- Повторный визуальный аудит по «Must Match Before Go-Live» (закрыть аудит 003).
- **Дифф контента живого ikpk.su против discovery-снапшота (март 2026)**: повторный прогон scrape-скриптов по entities + сравнение — новые статьи/семинары/акции/цены за время работы над ребилдом переносятся в снапшот. Прогнать при старте этапа и повторить перед самым go-live (сайт-источник продолжает жить).
- Обновить расписание/новости/акции по данным заказчика; убрать прошедшие семинары; письменное подтверждение.
- docs/content-update.md: интерим-процесс «JSON → PR → авто-staging → ручной prod» (≤30 мин); ночной rebuild по cron как страховка свежести + фильтр прошедших семинаров по дате на build.
- Переписать migration-runbook.md и observability.md под VPS (убрать Netlify/Vercel), заполнить backup-IP старого сайта, зафиксировать единое число страниц (256).
- Перевести parity-тесты (compare.spec, PARITY_REMOTE) в snapshot-режим — после переключения DNS живой оригинал исчезнет.
- Снять baseline из Вебмастера/GSC/Метрики (доступы запрошены в этапе 0; **без них go-live не проводим**).

Критерий готовности: все бюджеты зелёные 3 прогона подряд; CSP в enforce без поломок; re-audit 003 закрыт; расписание подтверждено; runbook исполним пошагово; baseline позиций снят.

### Этап 6 — UAT и go-live — **2 дня активной работы + 14 дней наблюдения** *(критический путь)*

Задачи:

- UAT с заказчиком на staging по functional_parity_checklist.md → письменный sign-off (gate rebuild_plan.md 5.2).
- **DNS-предполётный чек**: инвентаризация зоны сверена (этап 0), меняем только A/AAAA-записи веба; MX/SPF/TXT/DKIM не трогаем; после переключения — контрольная отправка/приём почты на домене.
- За 24 ч: DNS TTL → 300s. Переключение DNS; старый сайт read-only, архив 90 дней, backup-IP зафиксирован.
- Post-switch: validate-urls.ts против прода, sitemap в Вебмастер/GSC, проверка Метрики и целей в первый час, TTFB p75 ≤300ms из Москвы/СПб (синтетика), сквозная тестовая заявка до CRM.
- 14 дней: ежедневный дашборд (404-логи Nginx, индексация, позиции по baseline, конверсионные цели); rollback-триггеры: 404 >5%, индексация −20% за 7 дней, поломка форм. Rollback = возврат DNS (минуты при TTL 300s).

Критерий готовности: 14 дней без срабатывания триггеров; позиции без просадки >10%; почта домена работает; первые реальные заявки зафиксированы в Bitrix24.
