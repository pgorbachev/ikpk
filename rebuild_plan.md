# План: Rebuild сайта ИКПК

## Цель
Заменить текущий ikpk.su (Next.js, Performance 44/100) на новый сайт с тем же контентом, дизайном (±), но оптимальным стеком для SEO, производительности и управления контентом.

## Стек

**Astro + Strapi CMS**
- Astro SSG для статических страниц, React islands для интерактива (формы, фильтры, слайдеры)
- Strapi — headless CMS: RBAC (роли admin/editor/viewer), API tokens, audit log на изменения контента
- Деплой: CDN (Vercel/Netlify) для фронта, VPS/Cloud для Strapi
- Если позже появится кабинетная логика — выделяем отдельный app-модуль (Next.js/React) без пересборки основного сайта

## Data Freshness (обновление контента)

```
Контент-менеджер → Strapi → webhook → Astro rebuild → CDN
                                       (2-5 мин)
```

- **Расписание, цены, акции** — при любом изменении в Strapi автоматический webhook trigger → incremental rebuild → deploy. SLA: изменение видно на сайте **≤ 5 минут**.
- **Статьи, страницы** — тот же механизм, но менее критично по времени.
- **Fallback:** ручной trigger rebuild через CI/CD dashboard.
- **Nightly rebuild** — дополнительно, как страховка от пропущенных webhook'ов.

> **Открытый вопрос:** достаточно ли SLA ≤ 5 минут для обновления расписания, или нужен real-time (тогда расписание → SSR/API route вместо SSG)?

## Архитектура
```
[Strapi CMS] ──API──→ [Astro build] ──deploy──→ [CDN]
     ↑                   ↑ React islands
  Контент-менеджер       (формы, фильтры)
```

**Блокер для всей разработки.** До начала фронтенда и импорта — согласованная модель данных.

**Сущности:**

|----------|-----------------|-------|

| Seminar | name, slug, description, full_text, image, price, duration, status (planned/not_planned) | → CourseGroup, → Teacher[], → ScheduleEntry[] |
| Teacher | name, slug, photo, bio | → Seminar[] (M2M) |
| ScheduleEntry | date_start, date_end, city, price, seminar | → Seminar |

**Результат:** схема в Strapi, документация связей, согласование с заказчиком.

### Этап 2. Import Pipeline

Отдельный идемпотентный pipeline (скрипт), который:

1. Читает структурированный дамп из этапа 0.2
2. Создаёт/обновляет записи в Strapi через API
3. Загружает медиафайлы, привязывает к записям
4. Валидирует:
   - Все slug уникальны
   - Все связи разрешены (семинар → курс → институт)
   - Все медиафайлы загружены (сравнение с исходным дампом)
   - Нет пустых обязательных полей
5. Генерирует отчёт:
   - Успешно импортировано: N записей
   - Потери: список записей, которые не удалось импортировать (с причиной)
   - Конфликты: дубли slug, битые связи, отсутствующие медиа
6. Повторный запуск перезаписывает данные (идемпотентность по `legacy_id` — стабильный ключ, привязанный к исходному URL страницы, не к slug)
7. Таблица `_redirect_aliases`: legacy_id → old_url → new_slug. Если slug меняется, запись не теряется — связь через legacy_id, а old_url сохраняется для 301-карты

**Результат:** заполненная Strapi с верифицированным контентом.

### Этап 4. SEO

### Этап 5. Оптимизация

- Изображения: автоконвертация WebP/AVIF (@astrojs/image), lazy loading, srcset, заданные dimensions
- Шрифты: preload, font-display: swap, subset
- CSS: scoped styles (Astro default), критический CSS инлайн
- JS: 0 KB по умолчанию, React islands только для интерактива (формы, фильтры)
- HTML: минификация при build
- Заголовки: Cache-Control с длинным max-age для статики, immutable для hashed assets

### Этап 5.1. Security & Compliance

- **Strapi:** RBAC (admin / editor / viewer), API tokens с минимальными правами, регулярные бэкапы БД (daily), проверенный restore drill
- **Security headers:** HSTS, X-Content-Type-Options, X-Frame-Options, Referrer-Policy, Permissions-Policy (все отсутствуют на текущем сайте)
- **Формы и PII:** все формы (подписка, заявки) передают данные по HTTPS, хранение PII только в Strapi (не в логах CDN)
- **Cookie/Consent:** если используются аналитические cookie (Яндекс.Метрика) — баннер согласия согласно 152-ФЗ
- **Audit log:** Strapi ведёт лог изменений контента (кто, когда, что изменил)

> **Открытый вопрос:** нужен ли юридический контур по персональным данным (152-ФЗ) сразу в MVP, или это отдельный этап?

### Этап 5.2. UAT / Content-QA (gate до go-live)

Формальный этап приёмки **до** переключения DNS:

1. **Контент-проверка:** заказчик проходит по всем разделам, сверяет с текущим сайтом
2. **Функциональный паритет:** проход по checklist из этапа 0.4, каждый пункт ✅/❌
3. **SEO-проверка:** crawl всех URL, проверка мета-тегов, schema, 301-редиректов
4. **Performance:** Lighthouse CI по 4 шаблонам, сверка с KPI
5. **Sign-off:** письменное подтверждение заказчика на go-live

Результат: документ «UAT sign-off» с датой и списком принятых/отклонённых пунктов. Без sign-off — go-live не происходит.

### Этап 6. Миграция + Observability

**6.1 Pre-launch**
- Полный crawl нового сайта (проверка всех URL, статус-кодов, мета-тегов)
- Проверка 301-редиректов по карте URL
- Проверка функционального паритета по checklist из этапа 0.4
- Lighthouse audit на ключевых шаблонах (цели — см. KPI ниже)

**6.2 Go-live**
- Переключение DNS на новый сайт
- Подключение Яндекс.Метрики (счётчик 39506315)
- Подача sitemap в Яндекс.Вебмастер и Google Search Console (если будут доступы)

**6.3 Post-launch monitoring**
- 404/5xx мониторинг: логи CDN, алерты на рост 404
- Crawl ошибки: GSC/Вебмастер (если доступы)
- Индексация: `site:ikpk.su` в Яндексе и Google — отслеживать динамику
- Performance: еженедельный Lighthouse по 4 ключевым шаблонам

**6.4 Rollback план**
- DNS переключается обратно на старый сервер (TTL заранее снизить до 300 сек)
- Старый сайт продолжает работать параллельно минимум 2 недели после go-live **в режиме read-only** (формы/подписки отключены или проксируются на новый backend)
- Все write-операции (формы заявок, подписки) с момента go-live идут **только через новый сайт** — единый write-path, даже если DNS ещё не прорезолвился у всех
- Критерий отката: рост 404 > 5% от всех запросов ИЛИ потеря индексации > 20% страниц за 7 дней

> **Открытый вопрос:** есть ли write-функции (заявки, оплаты, подписки), которые должны сохраняться без потерь во время cutover? Если да — нужен единый API для приёма заявок, на который проксируют оба сайта.

## Критерии приёмки (KPI)

Измеряем на 4 шаблонах: главная, страница курса (depth=2), семинар (depth=3), статьи.
Методика: Lighthouse CI в Docker (фиксированное окружение), **медиана из 5 прогонов**, mobile preset (Moto G Power, 4G).
Для TTFB: synthetic monitoring из 2 точек (Москва + СПб), p75 за сутки.

| Метрика | Текущее значение | Целевое значение | Методика измерения |
|---------|-----------------|------------------|--------------------|
| Lighthouse Performance (mobile) | 36-56 | **≥ 85** | Lighthouse CI, медиана 5 прогонов |
| Lighthouse Performance (desktop) | 50-78 | **≥ 90** | Lighthouse CI, медиана 5 прогонов |
| Lighthouse SEO | 92 | **≥ 95** | Lighthouse CI |
| LCP mobile | 8.6-9.8s | **≤ 2.5s** | Lighthouse CI |
| TBT mobile | 290-2070ms | **≤ 200ms** | Lighthouse CI |
| CLS | 0-0.045 | **≤ 0.1** | Lighthouse CI |
| TTFB | 360-1130ms | **≤ 300ms p75** | Synthetic, Москва + СПб |
| Все URL из sitemap → 200 | ✅ | ✅ | Автоматический crawl |
| 301-редиректы | N/A | 100% покрытие карты URL | Скрипт проверки |
| Функциональный паритет | N/A | 100% checklist | UAT sign-off |

> HTML weight и JS bundle — ориентиры для разработки, но **не являются формальными критериями приёмки** (ориентир: HTML ≤ 100 KB, JS ≤ 50 KB для islands).
