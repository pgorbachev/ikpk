# IKPK Rebuild — Agent Guide

## Назначение проекта

Репозиторий содержит rebuild сайта `ikpk.su`:
- `web/` — новый фронтенд на Astro
- `cms/` — Strapi CMS
- `scripts/` — импорт контента и валидация URL
- `discovery/`, `docs/` — артефакты discovery, планов, проверки KPI и миграции

Цель: запустить новый сайт с функциональным паритетом, улучшенной производительностью и корректной SEO-базой без потери контента.

## Текущая фаза

Фаза: **visual/content parity hardening** (после первичного rebuild).

Текущий рабочий план: `docs/plans/002-parity-review-v2.md` (v2.1).
Приоритеты:
1. Очистка legacy HTML (`cleanBodyHtml`) и восстановление утраченных блоков главной.
2. Компонентный рендер статей/семинаров/статических страниц.
3. Тесты визуального паритета и build-gate.

## Source of Truth

Перед началом задач сверяться с документами в таком порядке:
1. `docs/plans/002-parity-review-v2.md` — текущая декомпозиция работ.
2. `rebuild_plan.md` — общая архитектура и этапы.
3. `discovery/functional_parity_checklist.md` — критерии паритета (go-live gate).
4. `docs/kpi-validation.md` — KPI и методика проверки.
5. `docs/migration-runbook.md` + `docs/observability.md` — миграция и мониторинг.

Если документы конфликтуют, приоритет у более узкого и более нового артефакта (plan > rebuild_plan).

## Правила выполнения

- Работать итеративно: маленькие изменения, затем локальная проверка.
- Не тянуть legacy CSS-модули с хешами в новый UI; использовать компонентный rebuild.
- Для контентных страниц сначала нормализация HTML, потом рендер компонентами.
- Любые SEO/маршрутизационные правки должны учитывать будущую миграцию и карту URL.
- Новые зависимости добавлять только при явной необходимости.

## Definition of Done

Задача считается закрытой, если:
1. Изменения реализованы в коде.
2. Локальный `build` проходит без ошибок.
3. Добавлены/обновлены тесты там, где это критично по рискам.
4. Отражено влияние на паритет (что стало лучше относительно production-версии).
