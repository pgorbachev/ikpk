# Постановка задачи: технический и SEO-аудит IKPK

## Роль
Сделать независимый технический и SEO-аудит сайта `ikpk.su` (и связанных доменов), чтобы принять решение: можно ли восстановить рост без полного редизайна, или нужен rebuild.

## Контекст
- Есть просадка поискового трафика (ориентир: после октября 2024).
- Есть коммерческое предложение от другого вендора на дорогую переделку.
- Нужна объективная картина рисков и план действий.
- На старте, вероятно, нет исходников и админ-доступов (black-box режим).

## Объекты аудита
- Основной: `https://ikpk.su`
- Связанные: `https://medshop.ikpk.su`, `https://kinezio.shop`, `https://ikpk812.ru`
- Опционально: `https://mudriydoctor.ru` как отдельный актив

## Цели
1. Найти технические причины, которые ограничивают SEO.
2. Разделить проблемы на критичные/средние/низкие с бизнес-влиянием.
3. Дать план исправлений на 90 дней с приоритетами, трудозатратами и ожидаемым эффектом.
4. Ответить, можно ли отложить полный редизайн и на какой срок.

## Формат работы
- Этап 1 (обязательный): Black-box аудит без доступов.
- Этап 2 (если будут доступы): проверка гипотез по Метрике/Вебмастеру/GSC/логам/админке.

## Что проверить (минимум)
- Индексация: `robots.txt`, `sitemap.xml`, статусные коды, `noindex/nofollow`, canonical, дубли, пагинация.
- Техчасть: редиректы, цепочки/циклы, `404/soft404`, mixed content, staging-ссылки, битые ссылки.
- Структура: информационная архитектура, внутренняя перелинковка, глубина клика, сиротские страницы.
- Контент/мета: `title/description/H1`, повторяемость, шаблонные страницы, thin content.
- Микроразметка: корректность `schema.org` / `JSON-LD` на ключевых типах страниц.
- Производительность: Core Web Vitals / базовые метрики рендера (desktop + mobile, выборочно по шаблонам).
- Мультидоменность: как распределяется SEO-вес между `ikpk.su` и связанными доменами.
- Конкурентный срез: 3-5 конкурентов по ключевым кластерам и разрыв по структуре/контенту.

## Ожидаемые артефакты
1. Executive summary на 1 страницу:
   - главные проблемы,
   - что чинить в первую очередь,
   - нужен ли срочный rebuild.
2. Таблица проблем:
   - URL/паттерн,
   - описание,
   - severity (`Critical/High/Medium/Low`),
   - влияние (`трафик/лиды/конверсия`),
   - сложность (`S/M/L`),
   - оценка трудозатрат (часы/дни),
   - способ проверки после фикса.
3. План `0-30 / 31-60 / 61-90` дней:
   - конкретные задачи,
   - ожидаемый эффект,
   - зависимости.
4. Отдельный блок: "Что невозможно доказать без доступов" и список нужных доступов.

## Критерии качества результата
- Каждый вывод подтвержден фактами (URL, ответ сервера, HTML-фрагмент, скрин/лог).
- Нет общих фраз без проверяемых примеров.
- Есть приоритизация по impact/effort, а не просто длинный список ошибок.
- В финале есть четкая рекомендация:
  - "ремонтируем текущий сайт" или
  - "готовим rebuild", и почему.

## Формат сдачи
- Отчет в Markdown (структурированный, с таблицами).
- Отдельно короткая версия для заказчика (без тех-жаргона, 1-2 страницы).
